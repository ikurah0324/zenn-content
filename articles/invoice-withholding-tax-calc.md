フリーランス向けの請求書作成ツールを個人開発していて、源泉徴収税の自動計算を実装しました。「報酬に10.21%を掛けるだけでしょ」と思っていたら、境界値・端数処理・浮動小数点と、考えることが意外とありました。この記事はその実装メモです。

## ルールのおさらい（根拠: 国税庁 No.2792）

原稿料やデザイン料など、個人への報酬・料金は支払側が源泉徴収します。税率は2段階です。

- **100万円以下の部分**: 10.21%（所得税10% + 復興特別所得税0.21%）
- **100万円を超える部分**: 20.42%（所得税20% + 復興特別所得税0.42%）
- 端数は **1円未満切捨て**
- 請求書に消費税が区分記載されていれば、**税抜金額をベース**に計算してよい

100万円超のときの計算式はこうなります。

```
源泉徴収税額 = 102,100円 + (報酬額 - 1,000,000円) × 20.42%
```

102,100円は「100万円 × 10.21%」の固定部分です。

## 実装

純関数ひとつに閉じ込めました。

```typescript
export const WITHHOLDING_THRESHOLD = 1_000_000;
export const WITHHOLDING_RATE_LOW = 0.1021;
export const WITHHOLDING_RATE_HIGH = 0.2042;

export function calcWithholdingTax(amountExTax: number): number {
  if (amountExTax <= 0) return 0;

  if (amountExTax <= WITHHOLDING_THRESHOLD) {
    return Math.floor(amountExTax * WITHHOLDING_RATE_LOW);
  } else {
    const low = Math.floor(WITHHOLDING_THRESHOLD * WITHHOLDING_RATE_LOW); // = 102,100
    const high = Math.floor((amountExTax - WITHHOLDING_THRESHOLD) * WITHHOLDING_RATE_HIGH);
    return low + high;
  }
}
```

ポイントは端数処理の位置です。低い段と高い段を**それぞれ切り捨ててから合算**しています。

## 計算例

| 税抜報酬 | 源泉徴収税額 |
|---------:|-------------:|
| 50,000円 | 5,105円 |
| 333,333円 | 34,033円 |
| 500,000円 | 51,050円 |
| 1,000,000円 | 102,100円 |
| 1,000,001円 | **102,100円** |
| 1,500,000円 | 204,200円 |
| 2,000,000円 | 306,300円 |

面白いのは **1,000,001円** のケース。超過分1円 × 20.42% = 0.2042円は切捨てで0円になるので、100万円ちょうどと税額が変わりません。境界値テストを書いていて「あれ、同じ値になるけどバグ？」と一瞬手が止まりましたが、これが正しい挙動です。

## 浮動小数点、大丈夫？

`0.1021` は2進浮動小数点で正確に表現できない数です。`Math.floor(x * 0.1021)` が、真の値のわずかに下に落ちた瞬間に **1円ズレ** ます。金額計算で1円のズレは信頼を失うレベルの事故です。

不安だったので、1円〜200万円の**全金額をBigIntの厳密な整数演算（×1021 ÷ 10000 の床関数）と突合**しました。

```javascript
for (let x = 1; x <= 2_000_000; x++) {
  const f = x <= 1_000_000
    ? Math.floor(x * 0.1021)
    : 102_100 + Math.floor((x - 1_000_000) * 0.2042);
  const exact = x <= 1_000_000
    ? Number(BigInt(x) * 1021n / 10000n)
    : 102_100 + Number(BigInt(x - 1_000_000) * 2042n / 10000n);
  if (f !== exact) console.log("mismatch:", x);
}
// → 不一致 0件
```

結果は**不一致ゼロ**。この税率・この金額レンジでは浮動小数点でも安全でした。

ただしこれは「たまたまこのレートでは大丈夫だった」という検証結果であって、一般論としては最初から `Math.floor(x * 1021 / 10000)` のように**整数を先に掛けてから割る**方が安全です（分子が2^53を超えない範囲で誤差なし）。私のコードも全数検証で担保しているとはいえ、次に書くなら整数演算にします。

## テストは境界値中心で

vitestで境界値を重点的に固めました。

- 0円・負の値 → 0円
- 100万円ちょうど → 102,100円（低い段の上限）
- 1,000,001円 → 102,100円（超過分の切捨て）
- 端数が出る金額（100,001円 → 10,210.1021 → 10,210円）→ 切捨て確認

「式が単純だからテスト不要」と思える計算ほど、境界に罠が潜んでいます。
