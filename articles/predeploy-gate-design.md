複数の個人開発プロダクトを並行運用していて、やらかしました。

**プロダクトAのビルド成果物を、プロダクトBの本番プロジェクトにデプロイした**のです。数分間、請求書作成ツールのURLでYouTubeリサーチツールが配信されていました。

原因は単純で、`wrangler pages deploy dist --project-name xxx` を実行したとき、シェルのカレントディレクトリが思っていた場所と違ったから。`dist` という相対パスはどこのリポジトリでも存在するので、wranglerは何の疑問もなく「そこにあるdist」をデプロイします。

幸いすぐ気づいて再デプロイで復旧しましたが、二度とやりたくないので、デプロイ前ゲート(`scripts/predeploy.sh`)を整備しました。この記事はその設計メモです。

## ゲートの全体像

デプロイは必ずこのスクリプトを通します。どれか1つでも失敗したら `exit 1` で止まり、デプロイコマンドは表示すらされません。

1. ユニットテスト(vitest)
2. 本番ビルド
3. E2Eスモークテスト(Playwright)
4. **dist/index.html のタイトルgrep(最終安全弁)**

## 最終安全弁: 「タイトルgrep」が一番効く

仕組みは拍子抜けするほど単純です。

```bash
EXPECTED_TITLE="サクッと請求書"
if grep -q "${EXPECTED_TITLE}" "${DIST_HTML}"; then
  echo "✓ dist/index.html に「${EXPECTED_TITLE}」を確認"
else
  echo "✗ 別製品のビルド成果物が混入している可能性があります。"
  exit 1
fi
```

「これからデプロイするdistは、本当にこの製品のものか?」を製品名で確認するだけ。冒頭の事故はこの3行があれば起きませんでした。テストが何百件通っていても、**デプロイ直前の成果物そのもの**を見ているチェックは別物です。

あわせて、スクリプト自身が冒頭で自分の場所からプロジェクトルートを解決するようにしておくと、どこから呼んでもカレントディレクトリ事故が起きません。

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
cd "${PROJECT_ROOT}"
```

## E2Eは「デモビルド」に対して流す

外部APIに依存するアプリのE2Eは不安定になりがちです。うちのYouTubeリサーチツールは、ビルド時に `VITE_API_BASE` を渡さないとモックデータで動く**デモモード**になる設計にしてあるので、E2Eはデモビルドに対して流しています。

```bash
# 本番ビルドを退避してからデモビルドを生成
cp -r dist dist_prod_backup
env -u VITE_API_BASE npm run build   # デモモード強制
npx playwright test
# E2E後、成否にかかわらず本番ビルドを復元
rm -rf dist && mv dist_prod_backup dist
```

ポイントは2つあります。

- `env -u VITE_API_BASE` で環境変数を明示的に除外する。シェルに残った値をViteが拾う事故を防ぐため
- E2Eが失敗しても**必ず本番ビルドをdistに復元**する。次のステップのタイトルgrepが「本番ビルド」を検証できるように

外部API非依存なのでE2Eがflakyにならず、「たまに落ちるから再実行」という習慣(=ゲートの形骸化)を防げます。

## E2Eのセレクタはスタイルクラスで引かない

ゲートを整備した直後、デザイン全面刷新でE2Eが1件落ちました。原因はテストがこう書かれていたこと。

```typescript
page.locator('span.font-bold.text-slate-500').filter({ hasText: '1' })
```

Tailwindのスタイルクラスをセレクタに使うと、再スタイルしただけで壊れます。`data-testid` に差し替えました。

```typescript
page.getByTestId('trending-rank').filter({ hasText: '1' })
```

落ちたこと自体はゲートが正しく機能した証拠で(機能は壊れていなかったがデプロイは止まった)、「E2Eのセレクタは data-testid / role / テキストで引く」という教訓が残りました。

## まとめ

- デプロイ事故は「テストが通っているか」ではなく「**今デプロイしようとしているものは何か**」の確認漏れで起きる
- タイトルgrepのような最終安全弁は3行で書けて、一番効く
- E2Eはデモモードに対して流すと安定し、ゲートが形骸化しない
- 相対パス+カレントディレクトリの組み合わせは個人開発でも事故る。スクリプトは自分の場所基準で動かす

個人開発は本番もレビュワーも自分だけなので、「自分を信用しない仕組み」を先に作っておくのが安上がりだと思います。
